# REPORT: pipeline-v1

## 1. Summary
Реализован единый orchestration-класс `TotoPipeline` для end-to-end запуска Toto-потока: загрузка тиража, подготовка матчей, генерация купонов, оценка результата и возврат итогового payload.
Добавлено сохранение результата последнего запуска в `data/last_run.json`.
Добавлено логирование ключевых метрик запуска (`draw_id`, `ROI`, `max_hits`).
Добавлены unit-тесты для проверки happy-path и извлечения результатов из матчей.
Этап закрывает отсутствие единой точки запуска для готовых модулей `TotoAPI`, `TotoOptimizer`, `TotoBacktest`, `FinalDecisionEngine`.

## 2. Goal
Сделать единый pipeline-runner `TotoPipeline.run_draw(draw_id, mode)`, который выполняет весь цикл обработки тиража и возвращает стандартный итоговый объект.

## 3. Scope
В scope вошло:
- добавление файла `toto/pipeline.py` с классом `TotoPipeline`;
- интеграция существующих модулей API/optimizer/backtest/final-decision;
- сохранение `data/last_run.json`;
- добавление тестов `tests/test_pipeline.py`.

В scope не входило:
- изменение сетевого клиента и формата API ответа;
- изменение алгоритмов optimizer/backtest/decision/value.

Проверено:

- файл `last_run.json` записывается на диск.

## 12. Risks / Known Issues
- При отсутствии рыночных `odds` используются fallback fair-odds из вероятностей; это упрощение для склейки блоков и может ограничивать value-логику.
- Если в данных нет валидных результатов матчей, backtest вернет низкоинформативные метрики.

## 16. Critical Review
Assumptions:
- `pool_probs` являются основным источником вероятностей на этапе v1.
- При отсутствии `odds` допустим fallback на fair-odds.

Edge cases handled:
- пустые/неполные вероятности (нормализация);
- отсутствие `results` на уровне draw (fallback на `matches[].result`);
- отсутствие `final_bet` у FinalDecisionEngine (fallback на `decision`).

Potential weak points:
- качество финальных решений ограничено минимальными `features`.
- логика odds-fallback не отражает маржу/рынок.
