# REPORT: final-decision-engine-v1

## 1. Summary
Реализован `FinalDecisionEngine v1`, который объединяет результаты `DecisionEngine` и `ValueEngine` в единый итоговый выбор ставки с формализованными правилами разрешения конфликтов и учетом порога `EV`.

## 2. Goal
Сделать единый детерминированный финальный слой принятия решения, чтобы после получения `decision` и `value_bet` система возвращала один прозрачный результат `final_bet`.

## 3. Scope
В рамках этапа:
- добавлен новый модуль `core/decision/final_decision_engine.py`;
- добавлены unit-тесты `tests/test_final_decision_engine.py`;
- добавлен отчет по этапу.

Вне рамок этапа:
- изменение формул `ValueEngine`;
- изменение базовой логики `DecisionEngine`;
- изменение обучения моделей и фичей.

## 4. Files Changed
- `core/decision/final_decision_engine.py`
- `tests/test_final_decision_engine.py`
- `reports/2026-03-21_17-20_final-decision-engine-v1.md`

## 5. Detailed Changes
1. Добавлен класс `FinalDecisionEngine` с методом `decide(probs, features, odds)`.
2. Выполнена интеграция вызовов `DecisionEngine.decide()` и `ValueEngine.calculate()`.
3. Зафиксированы правила merge:
   - при `value_flag=False` итоговая ставка отключается (`final_bet=None`);
   - при совпадении `decision` и `value_bet` выбирается совпадающий исход;
   - при конфликте одиночных исходов действует порог `EV=0.10`;
   - для двойных исходов (`1X`, `X2`, `12`) приоритет у `value_bet`, если он входит в двойной исход.
4. Добавлено логирование ключевых полей решения: `decision`, `value_bet`, `conflict`, `final_bet`.

## 6. Public Contracts / Interfaces
`FinalDecisionEngine.decide(...)` возвращает структуру:
- `decision: str`
- `value_bet: str`
- `final_bet: str | None`
- `confidence: float`
- `EV: float`
- `value_flag: bool`

Контракты `DecisionEngine` и `ValueEngine` сохранены без изменений.

## 7. Database Changes
Изменений БД не выполнялось: схемы, миграции и SQL-слой не затронуты.

## 8. API Behavior
Внешние API-эндпоинты не менялись.
Изменение касается внутреннего orchestration-слоя: теперь итоговая ставка формируется централизованно в `FinalDecisionEngine`.

## 9. Business Logic Changes
Старое поведение: не было отдельного централизованного слоя финального merge decision/value.
Новое поведение: добавлен формальный алгоритм выбора `final_bet` по правилам конфликта и порогу `EV=0.10`.

## 10. Tests
Добавлены тесты на:
- сценарий совпадения `decision` и `value_bet`;
- конфликт одиночных исходов по обе стороны порога `EV=0.10`;
- поведение при `value_flag=False` (ставка не делается);
- поведение для двойных исходов.

## 11. Manual Verification
Проверено вручную:
- результат содержит ожидаемые поля контракта;
- при конфликте исходов ветвление соответствует спецификации;
- при выключенном `value_flag` итоговая ставка отсутствует.

## 12. Risks / Known Issues
- Жесткий порог `EV=0.10` может быть неоптимален при смене рыночного режима коэффициентов.
- При плохом качестве входных вероятностей merge-правила могут стабильно выбирать формально корректный, но финансово слабый исход.
- Ограниченная explainability итогового решения усложняет аудит спорных кейсов в проде.

## 13. Deviations from Spec
Существенных отклонений от целевой v1-спецификации merge-правил не зафиксировано.
